# Web UI 页面设计规范 v2

## 0. 文档定位

本版规范用于替代 v1 在后台工具页上的落地方式偏差。

v1 强调“去卡片化、去容器化”，适合纯列表型页面；但在 `/model-config`、`/scheduled-tasks`、`/webhooks` 这类**单个条目同时承载状态、指标、标签和操作**的页面中，完全去卡片会损失可读性。

因此 v2 的核心修正是：

- **页面去模块化**
- **实体可卡片化**
- **操作要降噪**
- **状态表达尽量图标化、开关化**

一句话总结：

> 页面本身不要像“套了很多盒子”，但页面里的核心实体可以是高密度卡片。

---

## 1. 总原则

### 1.1 页面融入整体，不做“大模块盒”

页面结构应直接贴合主内容区展开，不额外套一层“总容器卡片”去包住多个 section。

允许：

- 页面标题
- 顶部主操作按钮
- 紧接着进入内容区
- 内容区内部按自然顺序排列 section

不允许：

- 用一个大圆角框同时包住“供应商配置 + 使用配置”
- 用厚重边框把 section 再次模块化
- 为了“看起来完整”而增加无语义外层容器

### 1.2 装饰不是层级，间距才是层级

层级优先通过以下方式表达：

- 标题字重
- 块间距
- 文本透明度
- 内容密度变化

而不是通过以下方式硬造层级：

- 大量背景块
- 粗边框
- 无意义分割线
- 说明文案堆叠

### 1.3 无意义信息一律删掉

默认删除以下内容，除非它能直接帮助用户决策：

- “当前展示 X 个...”
- “已启用 X / Y”
- 解释页面用途的长说明
- 分区之间的装饰横线
- 重复状态文案

规则：

> 用户从卡片内容本身已经能读懂的信息，不要再在页面层重复说一次。

---

## 2. 页面骨架

## 2.1 顶部区域

页面顶部只保留：

- 页面标题
- 当前页面真正需要的全局操作

例如：

- `/model-config`：标题 + 刷新
- `/scheduled-tasks`：标题 + 新建任务 / 刷新
- `/webhooks`：标题 + 新建 webhook / 刷新

不要在顶部塞入：

- 统计看板
- 大段解释
- “引导性说明”
- 与列表内容重复的状态摘要

## 2.2 Section 结构

每个 section 的标准结构应为：

1. `section title`
2. `section-level action`
3. `content grid/list`

推荐形式：

```txt
供应商配置                         [新建供应商配置]
[card] [card]
[card] [card]
```

不推荐形式：

```txt
┌────────────────────────────────────┐
│ Provider Configurations            │
│ 维护供应商的 Base URL、密钥...      │
│ 当前展示 6 个，已启用 4 个          │
│ ---------------------------------- │
│ [card] [card]                      │
└────────────────────────────────────┘
```

## 2.3 分区间距

分区之间优先使用留白，不使用横线。

建议：

- 页面主 section 间距：`space-y-6` 到 `space-y-8`
- section 标题与内容区间距：`space-y-3`

只有在真实信息密度过高、用户难以分辨段落时，才允许补极轻分隔。

---

## 3. 卡片使用规则

## 3.1 什么时候必须用卡片

以下场景应优先使用卡片，而不是普通行列表：

- 每个条目有多个维度信息
- 每个条目有独立状态
- 每个条目有独立操作
- 每个条目需要承载 tags / metrics / 开关 / 更多菜单

典型页面：

- `/model-config`
- `/scheduled-tasks`
- `/webhooks`

## 3.2 什么时候不要用卡片

以下场景优先使用行列表：

- 条目只展示单层文本信息
- 操作极少
- 不需要指标区
- 不需要状态切换

典型参考：

- `/mcp` 的页面骨架
- 纯搜索结果列表
- 纯名称列表

## 3.3 卡片不是“小模块盒”

卡片只用于承载实体本身，不用于再包一层 section。

正确：

- 页面直接放一组供应商卡片

错误：

- 页面先放一个“供应商配置模块盒”，再在模块盒里放供应商卡片

---

## 4. 卡片结构

## 4.1 标准卡片结构

卡片结构统一为五段：

1. 左上主体信息：图标 / 名称 / 副标题
2. 右上操作区：`状态图标 + switch + more`
3. 中部指标区：2 列或 2xN 面板
4. 底部标签区：带图标 tag
5. 必要时补一行低权重 meta 信息

示意：

```txt
[Icon] 标题                      [Ping] [Switch] [⋯]
       provider / model

| 指标 A | 指标 B |

[tag] [tag] [tag]
meta...
```

## 4.2 卡片间距

卡片内部要紧凑，避免被拆成彼此无关的块。

建议：

- 头部内边距：`px-5 pt-5`
- 指标区与头部间距：`mt-3`
- tag 区与指标区间距：`mt-2` 到 `mt-2.5`
- meta 行文字：`text-[11px]`

规则：

> 同一张卡里的信息要看起来属于一个整体，而不是 4 个堆叠小模块。

## 4.3 视觉强度

卡片本体可以存在轻微边框和极弱阴影，但必须克制。

允许：

- 轻边框
- 轻阴影
- hover 时极轻上浮

不允许：

- 厚重投影
- 多层外发光
- 过强渐变背景
- 卡片内再套多个高对比小卡片

---

## 5. 操作设计

## 5.1 禁止底部操作条

卡片底部不能专门留一行给“编辑 / 删除 / 更多操作”。

原因：

- 浪费垂直空间
- 打断信息阅读
- 让每张卡显得过重

替代方案：

- 统一放到右上角
- 或收纳到 `more` 菜单

## 5.2 头部操作布局

右上角统一采用：

`状态图标 + switch + more`

要求：

- 三者必须同一行
- 垂直居中对齐
- 不允许 switch 与操作按钮重叠
- 操作项增多时必须进入 `more`

不允许：

- 一列竖着排 3 到 5 个图标按钮
- switch 单独一列，操作按钮再绝对定位一列

## 5.3 `more` 菜单风格

`more` 菜单必须极简。

规则：

- 菜单项统一中性灰
- icon 风格统一
- 不做五颜六色的语义染色
- hover 只做轻背景和轻文字加深

只有在真正危险操作被用户直接 hover 到时，才允许出现更强语义色。

## 5.4 编辑 / 删除的默认权重

默认状态下：

- 编辑图标：浅灰
- 删除图标：浅灰

只有在 hover 卡片或 hover 图标时：

- 编辑趋向蓝色
- 删除趋向红色

原则：

> 用户先看内容，再看操作；操作不应抢主视觉。

---

## 6. 状态表达

## 6.1 启用状态统一用 switch

“已启用 / 已停用”不再用文本 badge 表达，统一改成 switch。

适用对象：

- 供应商配置
- 使用配置
- 计划任务
- webhook
- 其他可直接启停的实体

如果有 switch，就不要再重复放一个“已启用”标签。

## 6.2 连通性 / 检测状态优先图标化

像 `ping`、健康检查、可达性检测这类状态，优先用图标按钮表达，而不是“连接正常”这类文字 badge。

状态映射建议：

- idle：脉冲 / 待检测图标
- pending：旋转 refresh
- success：check
- failed：alert / x

文字只应出现在：

- tooltip
- hover title
- 卡片底部一行极轻 meta 说明

## 6.3 不重复表达同一状态

同一个状态不能同时出现在：

- badge
- 文本
- tag
- icon
- switch

例如“启用状态”已经由 switch 表示，就不再需要“已启用”文字。

---

## 7. 指标区与标签区

## 7.1 指标必须收拢成面板

像“模型数 / 引用数”“范围 / 用途”这类字段，不要松散地横向排开。

应使用：

- 两列指标面板
- 或带垂直分隔的双列布局

目的是让视觉焦点收拢，避免中间留白过大。

## 7.2 Tag 必须带图标前缀

底部 tag 不允许只靠文字表达。

每个 tag 都应有前缀 icon，让用户即使不读文字，也能迅速感知关系类型。

适用内容：

- 模型数
- 使用配置
- 范围
- 用途
- 账号
- webhook 来源
- 供应商可用性

## 7.3 Tag 是辅助识别，不是第二指标区

tag 只做“类型识别”和“关系识别”，不要把底部 tags 做成第二层统计面板。

避免：

- 过长文案
- 太多颜色
- 重复上方已经展示过的核心数值

---

## 8. 空状态

空状态必须简短，结构固定：

1. icon
2. 一句标题
3. 如有必要，一个主操作按钮

不要在空状态里写教学文案或解释页面原理。

正确：

- `暂无供应商配置`
- `[新建第一个供应商配置]`

错误：

- “你还没有创建任何供应商配置。供应商配置用于维护 Base URL、API Key 和推荐模型，建议先创建...”

---

## 9. 新建 / 编辑路径

## 9.1 表单复杂时，必须拆出独立页面

若“新建”动作会引出中等以上复杂度表单，不应塞回列表页顶部，也不要和预设入口混在一起。

正确做法：

- 列表页只展示已有实体
- 新建进入独立页面或明确弹层
- 编辑进入独立页面或明确弹层

例如：

- 供应商配置：独立页面
- 使用配置：可用弹层，因为复杂度可控

## 9.2 不混合“选择预设”和“已创建实体”

“Provider 预设”属于创建入口，不属于已存在配置列表的一部分。

规则：

- 已创建的内容，只在列表区展示
- 新建方式，只放在新建流程里

---

## 10. 色彩与交互权重

## 10.1 颜色分配

默认只使用三层色彩语义：

- 灰阶：绝大多数内容
- accent：主操作、选中、正向焦点
- danger：仅 destructive hover / error

不要把菜单项、次要 icon、普通标签做成彩色拼盘。

## 10.2 Hover 策略

hover 只做三件事：

- 轻微提亮
- 轻微位移
- 次要操作显现

不要做：

- 明显缩放
- 强发光
- 高饱和变色

---

## 11. 适用页面模板

## 11.1 实体卡片页

适用：

- `/model-config`
- `/scheduled-tasks`
- `/webhooks`

结构：

```txt
页面标题 + 全局操作

分区标题 + 分区操作
卡片网格

分区标题 + 分区操作
卡片网格
```

## 11.2 轻骨架页

适用：

- `/mcp`
- 纯列表页
- 以搜索、筛选、查看为主的页面

结构：

```txt
页面标题 + 主操作
搜索 / 筛选
列表或轻卡片
```

重点不是“完全照抄 `/mcp` 的视觉”，而是学习它的页面骨架：

- 不额外套大容器
- 不写多余提示
- 让内容直接出现

---

## 12. 反模式清单

以下做法在 v2 中视为反模式：

- 页面顶部放统计看板
- 用一个大盒子包多个 section
- section 之间加无意义横线
- 卡片底部专门留一行操作按钮
- switch 旁边再竖着堆一列 icon 按钮
- `more` 菜单里使用五颜六色的 icon
- 同一状态同时用 switch + badge + 文本重复表达
- tag 只写文字不带图标
- 指标松散横排，导致中间空洞
- 预设入口与已创建实体混在一个列表区域

---

## 13. 落地检查清单

设计或开发后台页面时，提交前至少自检以下问题：

- 页面是否还套着无语义的大外框？
- section 是否只保留了标题、按钮和内容？
- 是否删除了“当前展示 X 个”之类弱价值提示？
- 卡片操作是否全部收敛到右上角？
- 是否使用了 `switch + more` 的统一结构？
- `more` 菜单是否足够中性、克制？
- tag 是否全部带图标？
- 指标是否被收拢进稳定布局？
- 是否避免了重复状态表达？
- 新建入口是否和已有实体列表分离？

如果以上任意一项回答为“否”，则页面尚未达到 v2 标准。
