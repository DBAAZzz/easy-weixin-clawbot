# RSS 订阅与任务中心设计

## 背景

当前系统已经具备以下基础能力：

- 多账号运行时与账号级管理
- 主动消息推送能力
- 定时任务能力
- Web 管理后台

用户希望在此基础上引入一套可持续运营的 RSS 能力，但本期目标不是做一个孤立的“RSS 拉取器”，而是做成一套可管理、可调度、可复用的产品能力：

- 先有统一的 RSS 订阅列表，支持新增、编辑、删除、启停、预览、测试抓取
- 在订阅列表基础上创建任务，按固定频率向指定账号推送内容
- 支持高频任务，例如每 5 分钟一次，也支持每小时或每天摘要
- 支持任务级静默时段，避免夜间或特定时段打扰
- 同一账号可绑定多个任务
- 同一条内容只要已成功推送给账号，就视为该账号已经看过，不重复推送

用户同时确认了以下边界：

- 当前产品中，一个账号只对应一个激活会话，因此任务绑定账号即可，不再单独要求用户选择会话
- 媒体资源本期不做复杂缓存，图片、视频等只保留 URL 和基础展示信息
- 如果一条新闻后续被原站更新，仍视为同一条内容，不重复推送
- Agent 读取采集数据并按需查询的能力不在本期实现范围内，只进入 roadmap

## 目标

- 提供一个统一的 RSS 订阅列表页面，作为所有 RSS 能力的入口
- 提供一个统一的任务中心页面，支持基于订阅源创建摘要任务和快讯任务
- 将 RSSHub 地址和连接配置收敛到系统设置中，而不是分散在每个订阅源上
- 建立统一采集池，避免任务执行时反复直接请求外部源
- 以账号级去重规则保证“只要发过就算看过”
- 让 RSS 任务在产品形态上尽可能复用现有定时任务的管理方式
- 为未来把采集中心开放给 Agent tool 查询预留清晰边界

## 非目标

- 本期不实现 Agent 自动读取采集池并参与问答
- 本期不实现面向 Agent 的 RSS 查询工具
- 本期不实现复杂媒体缓存、媒体下载、媒体转存或素材仓库
- 本期不实现“内容更新后再次推送”的二次通知策略
- 本期不引入会话级独立所有权或会话级独立状态；订阅与任务均归账号所有
- 本期不实现多种精细化推送模板编辑器
- 本期不引入除 RSS 外的新信息源类型，但产品模型需为后续 Webhook 等来源预留扩展位

## 推荐方案

本期采用“源目录 + 统一采集池 + 任务中心”的方案。

这个方案的核心思想是：

1. 先把“从哪里拿内容”和“什么时候发给谁”拆开
2. 所有订阅源统一先进入采集池
3. 所有任务都只消费采集池，不直接请求 RSSHub
4. 去重和已看状态以账号为单位管理，而不是以任务为单位管理

这样可以避免以下问题：

- 每个任务都直接拉源，导致重复请求、重复解析和状态不一致
- 多个任务引用同一源时重复推送同一条内容
- 后续要把采集池开放给 Agent 时，需要重新拆底层模型

产品上，用户只需要理解两层对象：

- `订阅源`
- `任务`

内部的采集池是支撑能力，不要求用户在一开始就理解全部细节。

## 产品结构

### RSS 订阅列表

RSS 订阅列表是本期的基石，用来回答“抓什么内容”。

每个订阅源应支持：

- 新增
- 编辑
- 删除
- 启停
- 测试抓取
- 预览最近内容
- 查看健康状态
- 查看最近成功抓取时间
- 查看被多少任务引用

本期支持两类源：

- `RSSHub 路由源`
- `普通 RSS URL 源`

其中：

- `RSSHub 路由源` 依赖系统设置中的统一 RSSHub Base URL
- `普通 RSS URL 源` 允许直接填写完整 feed 地址

列表页建议直接展示：

- 名称
- 类型
- 状态
- 最近抓取时间
- 最近错误摘要
- 引用任务数

`预览` 是该页面最重要的能力之一。用户新增源后，必须可以立即看到最近抓到的内容，确认这个源是否配置正确。

### 任务中心

任务中心回答“什么时候，把哪些内容，发给哪个账号”。

任务统一抽象为：

`触发时间` + `数据源选择` + `内容整理方式` + `账号绑定` + `静默规则`

本期任务类型只做两种：

- `摘要任务`
- `快讯任务`

统一能力包括：

- 绑定账号
- 选择一个或多个订阅源
- 选择频率
- 配置静默时段
- 暂停 / 恢复
- 立即执行一次
- 预览本次将发送的内容
- 查看执行历史

任务列表建议直接展示：

- 任务名称
- 任务类型
- 绑定账号
- 关联订阅源数量
- 频率
- 静默时段
- 状态
- 上次执行时间
- 下次执行时间
- 上次执行结果

### 系统设置

RSSHub 服务配置放在系统设置中统一管理，不允许分散到每个订阅源中反复填写。

本期至少需要支持：

- RSSHub Base URL
- 认证信息
- 请求超时
- 测试连接

这层配置回答的是“去哪里请求 RSSHub”，而订阅源配置回答的是“请求哪个 route 或 feed”。

## 采集池与内容生命周期

### 统一采集池

所有订阅源的新内容先进入统一采集池，再由任务中心消费。

采集池的产品职责：

- 承接 RSSHub 或普通 RSS 的原始内容
- 做基础标准化
- 承担账号级去重所需的统一内容标识
- 为任务预览、执行和历史追踪提供稳定数据来源

这里的“基础标准化”必须至少覆盖以下内容：

- HTML 净化与富文本清洗，剥离脚本标签、埋点图、异常广告片段和不可见垃圾内容
- 将复杂富文本尽量转化为纯文本或受控的基础 Markdown 片段
- 为后续任务执行准备受控长度的标题、描述、正文摘录等轻量字段，避免下游直接消费原始 HTML

对于超长内容，本期不允许让任务执行时直接把多篇长文原文或 HTML 整体交给模型处理。采集池在入库阶段应先完成清洗与长度收敛，必要时产出受控长度的基础摘要或正文摘录，供后续任务直接复用。

本期虽然不把采集池直接暴露给用户单独管理，但页面上的预览、任务执行和后续 roadmap 都依赖它存在。

### 生命周期管理

RSS 内容会持续增长，本期必须带上内容生命周期规则。

建议产品规则：

- 普通热数据默认只保留近 14 天
- 长周期或深度源可放宽到 30 天
- 已推送记录的保留时间应长于内容池本身，避免内容刚被清理就失去“是否看过”的判断基础

生命周期策略的目标不是做归档系统，而是保证：

- 内容查询保持稳定性能
- 任务判断“新内容”时不会因为历史过长而明显变慢
- 后续给 Agent 暴露查询工具时，默认查询窗口可控

## 去重与“已看过”规则

### 账号级去重

本期“看过”的定义已经确认：

- 某条内容只要成功推送给某个账号一次，就视为该账号已经看过

这意味着去重范围必须是账号级，而不是任务级。

行为规则如下：

- 同一账号下任意任务成功推送过的内容，其他任务都不再重复推送
- 同一任务多次运行，也不会重复推送已成功送达过的内容
- 内容生成成功但消息发送失败，不算看过
- 源内容后续被原站更新，仍视为同一条内容，不重新进入候选列表

### 内容指纹

去重不能依赖标题文本模糊判断，必须依赖统一的内容指纹。

建议优先顺序：

1. RSS 自带 `guid`
2. 规范化后的原文链接
3. 由标题和发布时间生成的兜底指纹

任务消费和账号级已看判断都应基于这个统一指纹，而不是基于“这段文字看起来像不像同一条新闻”。

## 订阅源健康状态

RSSHub 路由或外部 RSS 源并不总是稳定可用，因此源目录必须内建健康状态，而不是无限重试。

建议产品状态：

- `正常`
- `退避中`
- `异常`
- `停用`

建议规则：

- 连续失败 3 次，进入退避，降低抓取频率
- 连续失败 10 次，标记为异常，停止高频死循环重试
- 异常状态应在源列表中明确可见
- 用户仍可手动测试抓取，验证是否恢复

这样可以避免失效源持续消耗抓取资源，也为后续接入更多信息源保留统一健康管理模型。

## 内容与媒体展示规则

本期不做复杂媒体系统。

对图片、视频、音频等资源，统一采用轻量策略：

- 保留原始 URL
- 保留基础展示信息，例如封面、标题、描述、来源链接
- 不在本期默认下载、缓存或转存媒体文件

推送时不做复杂模板区分，采用一股脑输出的默认风格：

- 有文字则发文字
- 有封面则带封面
- 有原文链接则附带原文链接

这样既满足用户对“不要过度设计”的要求，也给后续媒体缓存留出扩展空间。

## 任务行为规则

### 频率

任务创建时应提供清晰的预设频率，而不是一开始就要求用户输入 cron。

本期建议预设：

- 每 5 分钟
- 每 15 分钟
- 每 30 分钟
- 每 1 小时
- 每天

同时保留高级模式，为后续自定义频率扩展预留入口。

### 新建任务的历史边界

新建任务必须有明确的历史边界，防止第一次执行时把采集池中的历史沉淀内容一次性全部推送出去。

规则如下：

- 新创建的任务，默认只处理任务创建时间之后新进入采集池的内容
- 任务首次运行时，不追溯发送采集池里已经存在的历史内容
- 用户在没有显式触发补历史能力的情况下，不会收到过去数天或数周的历史大礼包

这条规则同时适用于摘要任务和快讯任务。

### 静默时段

静默时段按任务配置，不做全局统一静默开关。

规则如下：

- 静默期间继续采集内容
- 静默期间不执行推送
- 静默结束后，将静默期间累积的新内容合并为一次补发
- 补发不要求引入额外的高精度唤醒调度；只需在静默结束后的第一次常规任务调度执行时触发即可
- 如果静默期间没有新内容，则不发空消息

这样既避免打扰，又不丢内容。

### 无内容时的行为

任务到点执行但没有命中新内容时：

- 不发送“暂无新消息”之类的空推送
- 只在任务历史中记录“本次无新内容”

### 单次推送上限

无论是摘要任务还是快讯任务，单次执行时都必须有明确的内容条数上限，防止因为异常源、长时间静默积压或源站批量补发而导致一次消息过长。

规则如下：

- 每次合并发送时，只选择有限数量的内容进入最终消息
- 工程实现时必须有硬性兜底，绝对禁止无限制拼接
- 超出上限的多余内容，本期默认直接标记为“已处理 / 已忽略”，从而触发账号级去重，不再留待后续任务继续滚雪球式积压
- 系统的默认取舍是宁可丢弃较旧内容，也不允许任务队列因为历史积压而永久性阻塞

本设计建议默认优先选择最新且最重要的有限条目，而不是试图一次塞完全部积压内容。

### 任务重叠

同一账号可以绑定多个任务，多个任务也可以引用相同订阅源。

系统不禁止这种配置，但必须在后台提示：

- 这些任务覆盖了相同订阅源
- 由于账号级去重，先成功发送的任务会优先消费新内容
- 后执行的任务可能没有新内容可发

这是预期行为，不应视为系统错误。

## 摘要任务与快讯任务

### 摘要任务

摘要任务适合较长观察窗口，例如每小时一次或每天数次。

摘要任务默认由 AI 介入生成最终输出，本期不单独引入新的“摘要模型”配置，默认直接复用账号当前配置好的 chat 模型。

行为规则：

- 每次执行时，收集自上次成功发送以来的未读新内容
- 将这些内容合并为一条消息发送
- 每个条目默认带上标题、摘要或描述、原文链接，以及可用的封面信息
- 如果没有新内容，则本次不发送
- 任务执行时只允许把采集池中已经清洗、截断和整理好的轻量文本送入模型，不允许直接把多篇原始长文或 HTML 扔给模型
- 若采集池中已经准备好基础摘要或正文摘录，摘要任务优先消费这些受控字段，再由 chat 模型做最终组织和表述

摘要任务的产品语义是“窗口汇总”，而不是“逐条快发”。

### 快讯任务

快讯任务适合高频轮询，例如每 5 分钟一次。

为了避免刷屏，快讯任务也不应默认变成“一条内容一条消息”的连续轰炸。

行为规则：

- 每次执行只查看短窗口内的新内容
- 若命中 1 条，则发送 1 条内容组成的消息
- 若命中多条，也合并成 1 条消息发送
- 若没有新内容，则不发空消息

快讯任务本期应优先保证时效和稳定性。默认可以采用规则化拼接方式发送，不要求每次都经过 AI 二次总结；若后续需要统一文风，再在后续迭代中扩展为可选 AI 整理。

因此，快讯任务和摘要任务的主要区别是时间窗口长短，而不是消息发送条数。

## 任务预览、立即执行与执行历史

### 任务预览

预览的定义应固定为：

- 如果当前立刻执行该任务，将会发送什么内容

预览必须尽可能复用真实执行逻辑，避免“预览能看、正式执行结果却不一致”。

### 立即执行

任务创建或编辑后，用户应可直接手动执行一次，用于验证：

- 源是否能抓到内容
- 去重是否生效
- 推送内容是否符合预期
- 静默时段外的行为是否正确

### 执行历史

任务历史至少应回答：

- 什么时候跑过
- 本次有没有命中新内容
- 本次是否发送成功
- 若失败，失败原因是什么

这是后续排查任务配置错误、源异常和推送问题的关键入口。

## 页面与交互

### RSS 订阅列表页

页面重点是“配置源”和“验证源”。

关键操作：

- 新增订阅源
- 编辑订阅源
- 删除订阅源
- 启停订阅源
- 测试抓取
- 预览最近内容

关键反馈：

- 最近成功抓取时间
- 最近失败信息
- 当前状态
- 被多少任务引用

### 任务列表页

页面重点是“管理发送行为”。

关键操作：

- 新建任务
- 编辑任务
- 预览
- 立即执行
- 暂停 / 恢复
- 删除任务
- 查看执行历史

关键反馈：

- 任务类型
- 账号
- 频率
- 静默时段
- 上次执行结果
- 下次执行时间

### 账号详情页视图

账号详情页中应提供一个任务视图，帮助用户快速回答：

- 这个账号绑定了哪些 RSS 任务
- 这些任务分别看哪些订阅源
- 最近推送是否正常

这样用户不需要只依赖全局任务页来理解账号侧的 RSS 运行状态。

## 与现有定时任务能力的关系

本期不建议把 RSS 做成一套完全独立的调度系统。

推荐的产品抽象是：

- 统一保留一个“任务中心”
- 任务的管理外壳一致
- 任务的执行器类型不同

对用户来说，任务中心统一提供：

- 创建
- 编辑
- 启停
- 预览
- 立即执行
- 历史查看

区别只在于任务真正执行时：

- 现有 AI 定时任务执行的是一个 prompt
- RSS 任务执行的是“从采集池取内容并组织发送”

这样能最大化复用现有定时能力的心智和管理界面。

## Roadmap

本期不实现 Agent 数据源接入，但应明确列入 roadmap。

后续方向：

- 将采集池作为统一数据中心的一部分
- 对外暴露查询型 tools
- 由 Agent 在合适的时机自主决定是否调用这些 tools
- 让 RSS、Webhook 等不同信息源统一沉淀到同一数据中心

此时的产品形态不再只是“RSS 功能”，而是“信息源目录 + 数据中心 + 任务分发 + Agent 查询”的统一能力。

## 验收标准

- 用户可以在后台维护一份统一的 RSS 订阅列表
- 用户可以为账号创建摘要任务或快讯任务
- 任务支持频率、静默时段、暂停恢复、预览、立即执行和历史查看
- 同一条内容对同一账号成功推送一次后，不再重复推送
- 源内容被更新后，默认不作为新内容再次推送
- 任务在无新内容时不会发送空消息
- 新创建任务不会回溯推送采集池中的历史沉淀内容
- 任务单次发送存在硬性上限，不会因为异常积压而无限制拼接
- 媒体资源以 URL 和基础信息形式参与预览和推送，不要求本地缓存
- 采集池会先完成 HTML 净化、富文本清洗和受控长度收敛，再提供给下游任务使用
- RSSHub 地址由系统设置统一配置，不需要每个订阅源重复填写
- 系统能对异常源显示明确健康状态，并避免无意义高频重试
- 摘要任务默认通过 AI 生成最终内容，且默认复用账号当前配置好的 chat 模型

## 已确认的决策

以下问题已经在本次设计阶段明确，不再作为待定项：

- 订阅与任务均归账号所有；UI 可以从账号相关视图进入，但不引入会话级独立所有权
- 一个账号只对应一个激活会话，因此任务只绑定账号
- “看过”定义为成功推送过
- 内容更新不触发重复推送
- Agent 数据源能力本期不做，后续通过 tools 方式接入
